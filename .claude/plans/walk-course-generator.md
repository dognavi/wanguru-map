## 前提・確定した設計判断（v1から継続）

1. **コース店数**: 最大4店・最低2店(ただしv2では#7により2店未満に切り詰わることもある。後述)
2. **候補が少ない場合の表示分岐**: 1店のみ→単体紹介、0店→正直に「見つからない」旨を表示（詳細は下記「表示分岐と文言」参照）
3. **検索地点の地図表示**: 店のピン(①②③)とは別デザインの「出発地点」マーカー
4. **同座標グループの判定**: 完全一致（lat/lngの完全一致）。実データ検証済みで37組=74店と確認済み
5. **ルーティングAPIは使わない**: 店同士は直線で結ぶ。外部アクセスはNominatim検索時のみ

## 設計上の補足判断（実装前にレビューしてほしい点）

- **重複除外のタイミング(v1から継続)**: `data/shops.json`全体を一度だけ重複除外 → その結果に対して半径絞り込み・自動拡張を行う。半径内の件数チェックを重複除外前に行うと「実質1地点を2店」と誤判定するバグを防ぐため。
- **Nominatimの識別情報について(決定済み・案A採用)**: レビュー指摘#8にあった「User-Agentを明示」は、ブラウザの`fetch`/`XMLHttpRequest`が仕様上JavaScriptから`User-Agent`ヘッダーを上書きできない（ブラウザが自動設定する値が常に優先される）ため実現不可能。Nominatim固有の制約ではなくブラウザのセキュリティ制限であり回避不可。
  - **採用する方式**: ブラウザが自動送信する`Referer`ヘッダー（GitHub PagesのURL）に委ねる。追加のコード実装は不要
  - `email`クエリパラメータの付与（案B）は不採用。理由: 公開リポジトリ・公開サイトに連絡先メールアドレスが埋め込まれ、スパム収集の的になることを避けるため
  - デバウンス・多重送信防止（#8）を徹底することで、そもそも大量アクセスしない設計になっており、Refererのみでも利用ポリシー上十分と判断
- **半径拡張の閾値(#7・決定済み)**: 店同士のホップ距離が「極端に遠い」と判定する閾値は **5km** に確定。Phase 1で実データ(`data/shops.json`)全2,085店を起点候補として一括生成し、ホップ距離の分布を確認した結果:
  - 都心(渋谷駅)ではホップの中央値1.05km、10km閾値はほぼ発火しない
  - 離島(石垣島)を含む一部では最大9.99km(片道約2時間)のホップが実在し、「散歩」の範囲を明らかに超えるコースが生成され得ることを確認
  - 5kmに下げた場合、course件数は1930→1767、single件数は155→318に変化。5km超のコースは「単体紹介」に切り替わるが店自体は紹介され、導線は失われない
  - 姉妹プロダクト「散歩危険度チェッカー」を作るチームとして、片道2時間超を「散歩コース」と呼ぶのは方針に反するとの判断で5kmに確定

**Leafletの読み込み方法（決定・Phase 2で対応）**: CDNではなく`vendor/leaflet/`にファイルを同梱する。理由:
- 「完全無料・アクセスが増えても止まらない」という必須条件上、CDN障害時に地図が表示されなくなるリスクを避ける
- `@latest`のようなバージョン固定なしのCDN参照は、Leaflet側の破壊的変更が予告なく反映されるリスクがある
- Leafletは数百KB程度で、同梱コストは`data/shops.json`(1.17MB)より小さい
- バージョンは**1.9.4**（安定版）に固定する
- OpenStreetMapの**タイル画像**自体は同梱できない(都度必要な地域を配信するため)ため、引き続き外部リクエストが発生する。これは「地図タイル」として必要な外部アクセスであり、CDN依存の排除とは別問題として許容する

## コース生成ロジック（`js/route.js`）の関数設計 v2

すべて「引数 → 値を返す」純粋関数。DOM操作・fetch・console出力を含まない。

| 関数 | 役割 | v2での変更点 |
|---|---|---|
| `sanitizeShops(shops)` | `lat`/`lng`がNaN・非有限・範囲外(緯度±90/経度±180)の店を除外 | **新規(#10)**。パイプラインの最初に適用 |
| `haversineDistanceKm(a, b)` | 2点間の直線距離 | 変更なし。テストに実測既知距離を追加(#6) |
| `dedupeByExactCoordinate(shops)` | 完全一致座標をグループ化し代表1件に絞る。除外された店は`duplicatesByRepId`に保持 | 変更なし。`duplicatesByRepId`が#2の「近くにこんな店も」の直接の情報源になることを明記 |
| `findShopsWithAutoExpandingRadius(origin, shops)` | 半径2km→5km→10kmで自動拡張しながら絞り込み、距離昇順・同点はid昇順でソートして返す | **タイブレークを追加(#3)** |
| `buildNearestNeighborCourse(origin, candidates, {maxStops:4, maxHopDistanceKm:5})` | **originを起点(第0点)として**、直前の点から最近傍(同点はid昇順)を順にたどる。次の候補までの距離が`maxHopDistanceKm`を超える場合はそこで打ち切る | **起点をorigin自身に統一(#1)・タイブレーク(#3)・ホップ距離閾値による打ち切り(#7)を追加**。戻り値は0〜4件になり得る |
| `estimateWalkingMinutes(distanceKm)` | 分速80mで徒歩時間概算 | 変更なし |
| `generateWalkCourse(origin, allShops)` | 上記を束ねるオーケストレーター | 下記参照 |

### `generateWalkCourse`のパイプライン v2

```
sanitizeShops
  → dedupeByExactCoordinate（representatives, duplicatesByRepId）
  → findShopsWithAutoExpandingRadius（距離順・タイブレーク済み）
  → buildNearestNeighborCourse（origin起点・タイブレーク・ホップ閾値打ち切り）
  → 各stopに duplicatesByRepId[stop.shop.id] を nearbyDuplicates として付与（#2）
  → 件数に応じて status を決定
```

### status分岐（v2で統合。新しいstatus種別は追加しない）

- `stops.length >= 2` → `status:'course'`（2〜4件。ホップ閾値打ち切りにより結果的に2〜3件で終わることもある）
- `stops.length === 1` → `status:'single'`（半径内が1店だった場合と、ホップ閾値により2件目以降が打ち切られた場合の両方をこの状態に統合する。ユーザーに見せる文言は同一でよいため、statusを分けない）
- `stops.length === 0` → `status:'none'`（半径10km以内に候補が無かった場合。理論上はホップ閾値によっても起こり得るが、閾値の初期値が最大探索半径と同じ10kmのため、実際にはほぼ発生しない。将来閾値を10km未満に調整した場合は発生し得る点に留意）

各stopには `shop`・`distanceFromPrevKm`・`cumulativeDistanceKm`・`nearbyDuplicates`(#2、無ければ空配列) を含める。

**完了条件（検証ゲート）**
- `node --test` を実行し、全テストが green になること
- 以下を含む具体的なテストケースを用意する:
  - `sanitizeShops`: `lat`/`lng`が`NaN`・範囲外の店が除外されることを確認（#10）
  - `haversineDistanceKm`: 東京駅(35.681236, 139.767125)⇄新宿駅(35.690921, 139.700258)が**約6.134km**、東京駅⇄横浜駅(35.465981, 139.622402)が**約27.280km**となることを検証（本計画作成時にPythonのハバーサイン実装(地球半径6371km)で検算済みの値）（#6）
  - `dedupeByExactCoordinate`: 実データで重複除外後の件数が `2085 - 37 = 2048` になることを検証
  - `findShopsWithAutoExpandingRadius`: 2km圏内0〜1件→5km→10kmへ拡張されること、返り値が距離昇順・同点id昇順でソートされていることを検証（#3）
  - `buildNearestNeighborCourse`:
    - 起点がorigin自身であること（店ではなく検索地点からの貪欲法になっていること）を検証（#1）
    - 距離が同点の候補が複数あるとき、常に同じ順序（id昇順）で選ばれることを検証（#3）
    - 候補が5件以上あっても4件で打ち切られることを検証
    - ホップ距離が`maxHopDistanceKm`を超える候補には接続せず、そこで打ち切ることを検証（#7）
  - `generateWalkCourse`: 候補0件→`status:'none'`、1件→`status:'single'`、2件以上→`status:'course'`、ホップ閾値超過で1件に切り詰められるケース→`status:'single'`になることを検証
  - 各stopに正しい`nearbyDuplicates`（同座標グループの除外店のみ）が付与されることを検証（#2）
- **破壊テスト**（テストが実際にバグを検出できることの担保。それぞれ壊した状態でテストがfailすることを確認してから正しい実装に戻す）:
  - 起点をorigin以外（例: 最初の店）に戻す破壊 → 該当テストがfailすること（#1）
  - タイブレークを外す破壊（同点候補の順序が不定・入力順依存になる） → 該当テストがfailすること（#3）
  - 重複除外と半径絞り込みの順序を入れ替える破壊 → 実データ統合テストがfailすること
  - ホップ距離閾値チェックを外す破壊 → 該当テストがfailすること（#7）

### Phase 3: 入力UI（現在地取得・地名検索）
**作業内容**
- `js/geolocation.js`: 現在地取得ボタン + Geolocation API。エラーコード別に文言を出し分ける（#4）:
  - `PERMISSION_DENIED`(1) → 「位置情報の利用が許可されませんでした。地名で検索してください」
  - `POSITION_UNAVAILABLE`(2) → 「現在地を取得できませんでした。地名で検索してください」
  - `TIMEOUT`(3) → 「現在地の取得がタイムアウトしました。地名で検索してください」
  - Geolocationはsecure context必須。GitHub Pages(HTTPS)は問題なし、ローカル開発は`localhost`が例外的にsecure context扱いのため`python -m http.server`での確認は可能（LAN IP直アクセスは不可な点に注意）
- `js/nominatim.js`: **Nominatim濫用対策を最優先で実装（#8・最重要の運用リスク）**:
  - 検索は「入力確定(Enterキー)またはボタン押下」時のみ実行。キー入力ごとの逐次検索は行わない
  - 日本語入力のIME変換確定Enter（`isComposing`中のEnter）で誤送信しないようにする
  - 多重送信防止: 前回のリクエストが完了するまで次のリクエストを送らない（送信中はボタンをdisabledにする、または`AbortController`で前回リクエストをキャンセルしてから送信）
  - タイムアウト処理: `AbortSignal.timeout()`等で一定時間応答が無ければ打ち切り、「時間内に見つかりませんでした」等を表示する
  - 識別情報については追加実装不要（Refererに委ねる方式を採用済み。「設計上の補足判断」参照）
  - クエリパラメータに`countrycodes=jp`・`accept-language=ja`を付与し、海外の同名地名を誤って拾わないようにする
  - 検索結果の`display_name`（Nominatimが返す正式地名）を画面に表示し、「本町」「中央」等の同名地名でユーザーが意図しない場所に飛んだ場合に気づけるようにする
  - 地名が見つからない場合: 「その地名が見つかりませんでした」と表示（#9）

- Claude Browserで地名（例:「渋谷駅」）を検索し、ネットワークタブでNominatimへのリクエスト・レスポンス（`countrycodes=jp`・`accept-language=ja`が付与されていること、`display_name`が返っていること）を確認

## 将来の検討事項（v1のスコープ外・タスク化はしない）

- **全件線形スキャンの性能（#5）**: v1の店舗数（2085件）であれば都度全件をハバーサイン計算しても実用上問題ない想定だが、データ件数が大幅に増えた場合は、緯度経度によるバウンディングボックスでの事前絞り込みや、グリッド分割・k-d木などの空間インデックス導入を検討する余地がある
- エリア・ジャンル絞り込み（`areas.json`の活用）

## 進め方について

Phase 1から順に1つずつ実装 → 検証ゲートの結果を提示 → ユーザー承認 → 次のPhaseへ、という段階実行を前提とする。Phaseをまたいで先回りして実装しない。
