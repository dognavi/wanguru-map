# CLAUDE.md

## プロジェクトの目的

わんグルマップは、犬同伴OK店（わんグルのデータ）をもとに、現在地や地名の周辺で犬と一緒に入れるお店を近い順に地図上に表示する静的サイト。サーバーサイド処理なし、GitHub Pagesでの公開を前提とする。

## 技術スタック / 主要ライブラリ

- HTML / CSS / JavaScript（バニラJS、フレームワーク・ビルドツールなし）
- Leaflet.js — 地図描画（`vendor/leaflet/`にバージョン1.9.4を同梱。CDNは使わない。理由は「近隣店舗検索方針」の「経路探索APIは使わない」と同じ「完全無料・アクセスが増えても止まらない」原則）
- OpenStreetMap — 地図タイル
- Nominatim — 地名検索（ジオコーディングAPI。使うのは検索時の地名→座標変換のみ）
- Geolocation API（ブラウザ標準）— 現在地取得
- Node.js — 開発・テスト時のみ使用（`node --test` で近隣店舗検索ロジックを単体テスト）。本番の配信物はNode不要な静的ファイルのみ
- 経路探索API（OSRM/GraphHopper等）は**使わない**（詳細は「近隣店舗検索方針」参照）

- `data/`配下のJSONは既存データであり、原則として書き換えない（スキーマは[README.md](README.md)参照）
- ビルドステップを持たないため、GitHub Pagesでそのまま配信できる構成を維持する
- `js/`内はロジックとUIを分離する（`js/route.js`に近隣店舗検索ロジック、`js/render.js`に地図・カードのDOM生成、`js/main.js`にイベントハンドリング）。テストは`js/route.test.js`・`js/render.test.js`のように対象ファイルと対にする

## 近隣店舗検索方針（確定設計）

### 経路探索APIは使わない（重要）

- OSRM / GraphHopper等の経路探索APIは使わない。理由は2つ: (1) 無料枠に上限があり、超えると停止または課金が発生する。(2) ブラウザから直接呼ぶ構成ではAPIキーが開発者ツールから丸見えになり、リポジトリを非公開にしても防げない。「アクセスが増えるほど止まる/費用がかかる」構造を避けるため、経路探索APIは採用しない
- 道路に沿った経路は引かない。距離はハバーサイン公式による直線距離で計算する
- 外部アクセスは「地名→座標変換（Nominatim）」の検索時1回のみ。それ以外は同梱JSON（`data/shops.json` / `data/areas.json`）を読むだけ

### 入力

- 「現在地から探す」ボタン（Geolocation API）
- または「地名・駅名を入力」欄 → Nominatimで座標化
  - Nominatimのリクエストは1秒1回程度に抑え、過度な連続リクエストを避ける

### 検索ロジック（`js/route.js`: `findNearbyShops`）

- 入力地点から半径2kmで犬同伴OK店を探す。0件なら5km→10km→20km→50km→100kmと自動で範囲を広げる。1件でも見つかればそこで打ち切る（緩い保証。無理に数を確保しにいかない）
- 近い順に最大20件を返す（数珠つなぎのコースは組まない）
- 同座標の店（地図埋め込みのコピペにより`data/shops.json`に37組=74店存在）は1店だけ採用し、残りは「近くにこんな店も」として別枠で紹介する
- 距離はハバーサイン公式による直線距離で計算。徒歩時間は直線距離 ÷ 分速80mの概算
- 徒歩時間は10km以内のときだけ表示し、10kmを超える場合は「徒歩圏外」とだけ表示して時間は出さない（`js/render.js`: `shouldShowWalkingTime`）
  - **この10kmという閾値は、半径階層`[2, 5, 10, 20, 50, 100]`の境界の一つ（10km）とちょうど一致している。そのため「1回の検索結果内で、時間表示ありの店と徒歩圏外の店が混在する」ことは構造的に起こらない**（ある検索の結果は、採用された階層の半径以下の店だけで構成されるため）。これは2つの独立した設計判断がたまたま同じ値を共有した偶然の一致であり、**どちらかの数値だけを変更すると、そのとき初めて混在パターンが発生しうる**。変更する際はこの前提が崩れることを踏まえる
- 距離・時間は「目安」であることをUI上で明示する（実際は道なりで直線距離より長くなるため）

### 地図表示（`js/render.js`: `renderShopsOnMap`）

- 出発地点+全ての店に①②③…の番号つきピンを表示する（店同士を結ぶ線は引かない）
- **`fitBounds`の対象は近い順の上位5件+出発地点のみに絞る**。理由: 全20件を対象にすると密集地（渋谷など）で引きすぎ（zoom13相当）になり、ピンのタップ不能率が45%に達することを実測で確認した。上位5件に絞るとズームイン相当（zoom14前後）になりタップ不能率が下がる。該当件数が5件未満の場合は全件が対象になるため、遠方で店が少ないケース（例: 羅臼町、最寄り店まで約85.8km）でも出発地点+その店を必ず収める安全な挙動が保たれる
- **`renderShopsOnMap`の冒頭で必ず`map.invalidateSize()`を呼ぶ**。理由: ページ読み込み直後などレイアウト確定前に地図が生成されると、Leafletが古いコンテナサイズをキャッシュしたまま`fitBounds`を計算し、ズームが異常に深くなることがある（実測: 本来zoom14で収まるはずがzoom18になった）。描画のたびに最新サイズへ同期してから計算させる

### 一覧⇔ピンの連動（`js/main.js`）

- `shop.id`で双方向に紐付ける
- ピンタップ → 該当カードをハイライトし、一覧をスクロール
- カードタップ → 地図をそのピンへパン（ズームは変えない。合わせたズームの文脈を保つため）し、ポップアップを開く
- ポップアップの中身は店名のみ（リンクは持たせない）。理由: ピンのタップ不能率が実測で一定残っている状況で、ポップアップ内にさらに小さいリンクを詰め込むと誤タップの事故率が上がる。詳細情報はカード側に集約し、リンクURL生成ロジックの二重管理を避ける

### 表示

- 各店カード: 番号・店名・ジャンル・最寄駅（`access`）・直線距離・約○分（目安、10km超は「徒歩圏外」）・「詳しくは→わんグル」（`url`へのリンク）・「Googleマップで開く」（補助リンク）

### 店舗座標の精度について（重要）

- `data/shops.json`の`lat`/`lng`は、わんグルの店舗ページに埋め込まれたGoogleマップの中心座標であり、店舗の正確なピン位置ではない。埋め込み作成時に地図をパンしていると、数十m〜数kmずれる場合がある
- Google Places APIを使えば正確な位置が取得できるが、有料・APIキー必須のため不採用（「完全無料」の原則に反する）
- 対処として、各カードに「Googleマップで開く」リンクと、ピンの位置がずれている場合がある旨の注意書きを設けている

### スコープ（v1で入れないもの）

- エリア絞り込み・ジャンル絞り込みはv1では入れない（`areas.json`は同梱のみ。後日の拡張用）
- `data/parks.json`は現在未使用（将来のエリア/ジャンル絞り込み等での活用を想定して同梱のみ）

### 設計の分離（テスト方針）

- 近隣店舗検索ロジック（距離計算・重複除外・範囲自動拡張）は純粋関数として切り出し、UI（DOM操作・地図描画）から分離する
- カード表示の判定ロジック（`shouldShowWalkingTime`など）も同様に純粋関数として切り出す
- シグネチャの目安: 「入力座標 + 店リスト」→「距離順の店リスト（最大20件）」を返す純粋関数
- テストは`node --test`（Node.js組み込みテストランナー）で実行する

## コーディング規約

- ES modules（`<script type="module">`）を基本とし、外部フレームワークは導入しない
- インデントは2スペース、文字コードはUTF-8、改行はLF（`.editorconfig`参照）
- DOM操作・地図初期化・データ取得（fetch）は関数単位で分離し、グローバル変数を増やさない

## よく使うコマンド

```bash
# ローカルプレビュー（fetchでJSONを読むため簡易サーバー経由で開く）
python -m http.server 8000
# または
npx serve .

# 近隣店舗検索ロジック・カード表示判定の単体テスト（js/route.js・js/render.jsの純粋関数が対象）
node --test
```

Lint/型チェックツールは未導入。近隣店舗検索ロジック（距離計算・範囲自動拡張等の純粋関数）は`node --test`でテストし、UI（地図描画・Geolocation・Nominatim検索・ピン⇔カード連動）はブラウザでの目視確認を基本とする。

## 注意事項

- `.env`は読まない・コミットしない（本プロジェクトはNominatim/OpenStreetMap/Geolocationのみを使い、APIキーは不要）
- Nominatimの利用ポリシー上、リクエストは1秒間に1回程度に抑え、過度な連続リクエストを避ける
- 地図・データの出典表示（OpenStreetMapの著作権表示、わんグル由来である旨）を画面上に残す
- 店舗座標の精度については「近隣店舗検索方針」内の「店舗座標の精度について」を参照
